# 项目执行路线图

## 1. 目标

围绕当前实际业务链路：

1. 客户订单录入与确认
2. 生成施工单
3. 工序拆分与任务执行
4. 质检与入库
5. 发货与签收
6. 发票、收款、对账闭环

本轮优化目标不是继续扩页面，而是补齐现有模块中的关键闭环，让系统更适合真实上线使用。

## 2. 执行原则

- 优先补闭环，不新增无必要页面。
- 优先复用后端已存在字段与接口。
- 每一阶段只做最小可用能力，避免过度设计。
- 每完成一个阶段，回写本文件进度并提交。

## 3. 当前核心缺口

### P0：资料与凭证无法形成闭环

当前后端已存在附件字段，但前端大多不可见、不可上传、不可追溯：

- 施工单 `design_file`
- 质检单 `attachment`
- 发票 `attachment`
- 发货单 `receiver_signature`

这会导致关键单据虽有数据结构，但业务无法在页面中直接查看、核对、留痕。

### P0：异常链路不完整

当前主流程覆盖较多，但异常处理仍偏弱：

- 审批驳回后的补料、补图、重提缺少明显引导
- 生产异常、质检不合格、发货拒收后缺少统一收口动作
- 页面提示偏“列表型”，缺少“下一步该做什么”的流程提示

### P1：批次与追溯能力不足

真实生产中常见追问：

- 这批货对应哪个客户订单、哪个施工单、哪个工序
- 哪次质检出的结论
- 哪次发货出的库
- 出问题时能否快速反查

当前系统已有部分字段，但前端链路仍偏散，跨模块追溯成本高。

### P1：交付到财务的闭环还不够紧

当前已有发票、收款、对账模块，但缺少更直接的业务联动提示：

- 发货后是否应提醒开票
- 已开票但未收款是否应提醒财务跟进
- 收款计划、实际收款、对账状态的串联还不够明显

### P2：权限与数据范围控制还需要收紧

真实落地后会遇到：

- 业务员只能看自己的客户订单
- 车间只能看本部门任务
- 财务只看财务数据
- 主管可跨部门查看但不可随意改关键单据

如果不收紧，系统可用但不够“上线级”。
